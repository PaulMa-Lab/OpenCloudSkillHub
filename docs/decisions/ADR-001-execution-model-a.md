# ADR-001：执行采用「模型 A」——Hub 只读/出计划，宿主执行 + 宿主审批

- 状态：**Accepted**（用户 2026-06-02 确认）
- 日期：2026-06-02
- 关联：架构文档 §0、§10；ADR-003（解耦，与本决定相互强化）

## Context

OpenCloudSkillHub 要让 Agent「学会并装配能力」，这必然涉及会改变用户机器的动作：装依赖、写文件、跑 shell、下模型、读本地文件、（可选）上传远程。

核心问题不是「OCR 怎么装」，而是：**真正改变用户机器的那条命令由谁执行？审批发生在哪一层？**

存在两种架构：
- **模型 A（顾问型）**：Hub 的 MCP server 只提供只读 resources 与只读/纯计算 tools（环境检测、生成安装计划、验证计划、诊断、写反馈）。真正的可变操作交给宿主 Agent（Claude Code 等）用它自己的工具执行，复用宿主已有的权限审批层。Hub 永不触碰用户机器可变状态。
- **模型 B（执行型）**：Hub 自己跑 `pip install` 等命令，并自建一套审批机制。

## Decision

**采用模型 A。** 具体约束：
1. Hub 所有 tool 设计为 `read_only` 或纯计算；不在 Hub 进程内执行任何会改变用户机器状态的操作。
2. 可变操作以**结构化安装计划**返回，每个 step 自带 `risk` / `approval_required` / `rollback`，由**宿主**在用户批准下执行。
3. 验证脚本、smoke test、runner 等 skill 资产由**宿主执行**，Hub 只返回「怎么跑 + 期望结果」。
4. 唯一允许的副作用：`submit_skill_feedback` 写 Hub 自己的本地日志（不碰用户机器）；可选的远程推理能力默认关闭且不属于平台核心（见 ADR-003）。

## Consequences

**正面**
- 审批复用宿主已被用户信任的权限 UI，**不重复造审批轮子**，不扩大攻击面。
- 安全边界干净：Hub = 知识 + 判断，宿主 = 执行 + 审批。Hub 被攻击/guide 写错的最坏后果是「给了一个坏计划」，而非「直接在用户机器上跑了坏命令」。
- 与 ADR-003 相互强化：核心连 skill 代码都不跑，天然 skill-agnostic。

**负面 / 代价**
- 「最后一公里」依赖宿主忠实执行计划。无 shell 能力的纯聊天宿主只能读 guide、看计划，无法真正落地安装——可接受（那类宿主本就不该自动装东西）。
- Hub 无法用代码做「聪明的自动安装/自愈」，这些智力被推给「Agent 读 guide + 宿主执行」。MVP 以 Claude Code 这类有 shell 的宿主为目标。

## Alternatives considered
- **模型 B（执行型）**：更「一体化」，但要在 MCP 层重造审批（MCP 无服务端弹审批框的标准原语），且让 Hub 直接持有改用户机器的能力，安全风险与维护成本都更高。**否决。**

## Uncertainty
- 不同宿主对「逐步审批」「按计划执行」的支持程度不一，需里程碑 5 真机观察。若宿主不肯按计划走，体验会退化为人工介入——这是范式验证的真实风险点，不掩盖。
