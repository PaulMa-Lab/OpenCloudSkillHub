# ADR-004：课程贡献是供应链问题——自动校验把关 + 信任分级 + 不自动发布

- 状态：**Accepted**（用户 2026-06-02 提出未来需求时确立原则）
- 日期：2026-06-02
- 关联：ADR-001（模型 A，运行期安全网）、ADR-003（解耦，贡献不改核心）；课程包契约、领域注册契约
- 性质：**面向未来的原则约束**。贡献流水线本身是第二阶段，但其安全边界现在就定，避免 M3/M5 挖坑。

## Context

长期目标：支持多个用户 / Agent 向学校贡献并维护自己的课程（skill package），并对课程做（部分）自动审核。

关键事实：课程包内含**可执行资产**（`verify_*.py`、smoke test、runner 模板），这些资产最终由**宿主在用户机器上执行**（模型 A）。因此：

> **接受外部课程 = 接受外部代码进入一条"会在用户机器上运行"的渠道。课程贡献本质是供应链安全问题，是本项目最敏感的面。**

"自动审核"这个词容易让人以为可以全自动放行。但**可执行资产的良性无法靠静态校验证明**（停机问题级别的根本限制）。

## Decision（原则，非实现）

1. **自动校验是第一道闸，但"通过校验" ≠ "可信"。**
   `validate_skill_package` / `validate_domain` 只能证明"结构合规 + 安全声明自洽 + 引用存在 + 已知危险模式扫描"。它**证明不了**脚本不是恶意的。绝不把"校验通过"等同于"安全可用"。

2. **信任分级，且 official 不可自动达成。**
   - 新贡献默认 `unverified`（trust.yaml 缺省值）。
   - 自动校验通过，**最多**进入 `community`（带告警标注）。
   - `official` 必须经人工 / 受信流程，**不可由自动流程授予**。
   - `trust` 不可自声明（已由 R-GOV-1 / D-GOV-1 强制）。

3. **自动流程不自动发布。**
   自动化只能把课程放进 quarantine / `unverified` 索引；**`recommend_learning_path` 与 catalog 必须标注 trust**，且 `unverified` 默认不进入主动推荐（或必须显式告警"未审核，自担风险"）。

4. **运行期安全网始终是模型 A。**
   贡献课程的任何脚本都由**宿主在用户显式批准下执行**，Hub 永不执行课程代码。这是兜底：即便坏课程进了索引，没有用户批准也跑不起来；用户批准时也应被清楚告知"这是 unverified 来源的脚本"。

5. **来源与凭证（provenance）是审核与撤回的锚。**
   课程应携带可核验来源（`source` repo / maintainer / 内容 hash / 未来的签名）。审核、版本治理、撤回（takedown）都以 provenance 为锚。

## Roadmap（第二阶段，现在不做）

`submit_skill_package`（上传）→ 沙箱内校验 → quarantine（unverified）→ 人工/自动审核 → trust 提升 → 索引。
领域系统在线注册同理（endpoint 信任 + 审核）。评分 / 社区治理 / 撤回流程随后。

## 现在就要保留的 hooks（让 M3/M5 不挖坑）

- 保持 `trust` 默认 `unverified`、validator 作为**唯一收录闸**、manifest `source` 字段位（已有）作 provenance 预留。
- `skills://catalog` / `domains://catalog` / `recommend_learning_path` 已携带 `trust` 与 `validation` 字段——**继续保留，并在推荐逻辑中真正使用**（unverified 降权/告警）。
- 课程资产执行**始终走宿主 + 用户批准（模型 A）**；绝不为"方便"在 Hub 侧执行任何课程脚本。
- 校验器规则按 `code` 稳定演进；新增"危险模式扫描"（如 verify 脚本里出现网络/子进程/写系统路径）时作为新 `R-*` 规则增量加入，不破坏既有契约。

## Pushback（明确写下，防止将来被"自动审核"四个字带歪）

- **不要承诺"自动审核 = 安全"。** 自动化只能做"形状 / 安全声明 / 已知危险模式扫描"。真正的信任要么靠 **provenance + 人工**，要么靠**运行期沙箱 + 用户批准**。两者都不能被一个静态校验器替代。
- **不要做"自动晋升到 official"。** 这是把供应链信任拱手让给一个可被绕过的检查器。
- **不要在 Hub 侧执行课程脚本来"预审"**，那等于把供应链攻击面从用户机器搬进平台。若将来要动态预审，必须在隔离沙箱里、且明确这只是启发式而非保证。

## Uncertainty
- "危险模式扫描"能覆盖到什么程度尚未验证；它能提高门槛，但永远不是充分条件。
- 沙箱化动态预审的成本/收益要到第二阶段才评估，现在不投入。
