# OpenCloudSkillHub · 学习方法（school://curriculum）

这是你在这里学习**任何一门课**的通用方法。它不绑定任何具体能力——下面用 OCR 举例，但同样适用于未来的浏览器自动化、表格分析、PDF 抽取等课程。

学习一门课是一条有**关卡（gate）**的流水线：**没通过当前关卡的判据，就不要进入下一阶段。** 这能防止你在环境不达标时硬装、在没装好时就调用、在失败时盲目前进。

---

## 阶段 0 · 定向（Orient）
- **读**：`system://guide`（总则）→ 本文。
- **做**：内化三条总则（Hub 不执行你执行 / 危险动作先批准 / 最小授权失败即停）。
- **进入下一步的判据**：你清楚「平台只给知识和计划，执行与审批由你和用户负责」。

## 阶段 1 · 选课（Select）
- **读**：`skills://catalog`；用 `search_skills(task)` / `recommend_learning_path(task)`（后者同时给出候选领域系统与通用技能）。
- **做**：把用户任务映射到一门或几门课。
- **判据**：命中一个候选课程的 `id`。
- **失败/无命中**：如实告诉用户「目前没有匹配的课程」，并用 `submit_skill_feedback` 回传「缺这类能力」。**不要假装有能力。**

## 阶段 2 · 评估匹配（Assess fit）—— 关键关卡
这一步决定「该不该学这门课」，比「怎么学」更重要。
- **读**：`skill://<id>/guide`（尤其 when-to-use 部分）+ `skill://<id>/safety`。
- **做**：
  1. 确认任务**真的需要**这门课，且没有更简单的替代（例：PDF 有文本层就直接抽，不必上 OCR）。
  2. 调 `detect_environment(skill_id)`，对照课程前置（Python/磁盘/网络/平台等）。
  3. 读 safety，记下这门课会申请哪些宿主权力（`tools_required`）、哪些步骤危险。
- **判据**：任务匹配 ✅ + 环境达标 ✅ + 你已知道危险点 ✅。
- **失败**：环境不达标 → 报告障碍（缺什么、要用户做什么），不要硬装；任务其实不需要 → 选更简单的路径并说明。

## 阶段 3 · 制定计划（Plan）
- **读**：`skill://<id>/install_<platform>`（如 `install_windows`）。
- **做**：调 `generate_install_plan(skill_id, {profile_id?})`。课程可能提供多个安装变体（profile），**根据 guide 的取舍说明选一个**（领域取舍由你推理，平台不替你选）。
- **判据**：你拿到一份分步计划，每步带 `risk` / `approval_required` / `rollback` / 预计下载体积。
- **注意**：此刻**还没有执行任何东西**，只是有了计划。

## 阶段 4 · 申请批准（Approve）—— 安全关卡，不可跳过
- **做**：把计划中 `approval_required: true` 的步骤**逐条、用人话**讲给用户：要做什么、为什么、有何风险、可否回滚、要读哪些文件、要下多大模型。
- **判据**：用户对每个危险步骤明确表态。
- **规则**：用户批准哪些，你**只执行哪些**；未批准的不做；涉及数据外发（上传远程）默认禁止，必须单独显式批准。

## 阶段 5 · 执行（Execute）—— 你用自己的工具做
- **做**：按被批准的步骤，用你自己的 shell/文件工具执行：通常是「建独立 venv → 装钉版依赖 → 跑 smoke test → （批准后）下模型 → 生成可复用 runner」。
- **判据**：每一步成功再做下一步。
- **失败即停**：任一步失败 → **立即停止**，进入阶段 7（排错）。不要重试乱试、不要继续往后装。回滚通常 = 删掉该 venv 目录（因隔离，删了不伤系统）。

## 阶段 6 · 验证（Verify）—— 客观判据，不靠感觉
- **读/做**：调 `get_verification_plan(skill_id)` 拿到「验证脚本 + 怎么跑 + 期望输出」；**你执行**该验证脚本，把实际输出与期望比对；同时对照 `skill://<id>/verify` 的验收清单。
- **判据**：验证**客观通过**（如 sample 图识别出期望文本）才算装好。
- **失败**：进入阶段 7。

## 阶段 7 · 排错（Troubleshoot）
- **读**：`skill://<id>/troubleshooting`（症状→原因→对策）。
- **做**：把真实报错日志交给 `diagnose_error(skill_id, logs)` 做归因，按建议采取**单一明确**的下一步（如换 profile、调依赖版本）。
- **判据**：要么修复后回到阶段 5/6，要么判定「这条路走不通」。
- **走不通时**：经用户显式批准走兜底方案（若该课提供），或**回滚并如实报告失败原因**——不要假装成功。

## 阶段 8 · 使用（Use）
- **读**：`skill://<id>/recipes`（最小可用调用范式）。
- **做**：用已装好的 runner 处理用户的真实任务（读用户文件前仍需批准）。
- **判据**：交付正确结果。

## 阶段 9 · 反馈（Feedback）—— 别省略
- **做**：调 `submit_skill_feedback`，回传：成功 / 失败及原因 / 缺失能力 / **guide 与真实情况不符**。
- **为什么重要**：课程靠真实反馈迭代。尤其「guide 说的和实际做的不一致」是平台最该知道的信号。

---

## 贯穿所有阶段的安全原则
1. **Hub 给知识与计划，执行与审批是你和用户的责任**（不要期待平台替你跑命令）。
2. **危险动作先批准、最小授权、数据默认不外发。**
3. **失败即停 + 可回滚优于强行成功。**
4. **不确定就说不确定**，不要编造能力或谎报成功。
5. **优先更简单的解**——能不装就不装。

## 「学会了」的判定标准
当且仅当：① 验证客观通过；② 你能用这门课处理用户的真实任务；③ 全程没有未经批准的危险动作发生。三者同时满足，才算学会了这门课。
